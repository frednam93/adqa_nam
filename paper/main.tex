\begin{abstract}
This technical report describes our system for DCASE 2026 Task 5, Audio-Dependent Question Answering.
We build on the open-source Qwen3-Omni-30B-A3B-Instruct model and focus on reducing shortcut reliance on the textual question prior.
Our strongest development systems use LoRA supervised fine-tuning on audio-dependent training items, small proportions of empty-audio negative examples, and a text-only response normalizer for parse-failed model generations.
The best train-only system reaches 67.27\% development accuracy after response normalization, compared with 65.90\% for our local Qwen3-Omni baseline.
\end{abstract}

\section{Introduction}
DCASE 2026 Task 5 evaluates audio-dependent multiple-choice question answering.
A central difficulty of the task is that large audio-language models can answer some questions from textual priors without using the audio signal.
Our approach therefore keeps the model architecture simple and concentrates on post-training data selection and inference-time response normalization.

\section{System Overview}
All submitted systems are based on Qwen3-Omni-30B-A3B-Instruct.
We apply parameter-efficient LoRA fine-tuning with rank 4, alpha 8, dropout 0.05, and trainable query and value projections.
The audio tower and multimodal projector are frozen.
Inference uses greedy decoding with an instruction to output only the exact option text.

\section{Audio-Dependency-Aware Training Data}
We first run diagnostic ablations on the official training set using normal audio-question pairs, empty-audio questions, and shuffled-audio questions.
Items are grouped according to whether the base model appears to require the correct audio to answer correctly.
The main supervised fine-tuning subset uses the strong audio-dependent bucket and excludes easy text-prior samples.

\section{Empty-Audio Negative Examples}
To reduce unsupported guessing, we add a small number of empty-audio negative examples.
For these examples the audio input is removed and the target response is ``Cannot be determined from the audio.''
We sweep the negative ratio and find 5\% and 2.5\% to be the strongest settings among the tested systems.

\section{Response Normalization and Ensembling}
Raw model outputs are first parsed by exact option-text matching and simple prefix stripping.
For parse failures, selected submissions use the base Qwen3-Omni model as a text-only response normalizer over the candidate choices.
This normalizer is counted as an additional 30B model component in the metadata.
The ensemble submission combines two fine-tuned systems by voting; tied cases are resolved by the train+dev empty-5\% system.

\section{Submitted Systems}
We prepare four candidate submissions.
System 1 is the best train-only checkpoint with 5\% empty negatives.
System 2 retrains the same recipe on the union of train and development strong audio-dependent items.
System 3 uses the train+dev recipe with 2.5\% empty negatives and a longer 3000-step checkpoint.
System 4 ensembles Systems 2 and 3 with System 2 as the tie breaker.

\section{Discussion}
The improvements over the local Qwen3-Omni baseline are modest, but consistent with the goal of reducing text-prior shortcuts.
Experiments with RL-style continuation, category-balanced resampling, and shuffled-audio negatives did not outperform the simple strong-audio SFT recipe in our current runs.
The train+dev systems are intended only as final evaluation submissions because development labels are used during training.
